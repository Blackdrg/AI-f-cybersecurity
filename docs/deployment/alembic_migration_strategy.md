# Alembic Database Migration Strategy

## Overview

AI-f uses **Alembic** (SQLAlchemy's migration tool) for database schema versioning.
Migrations are sequential, immutable, and must be backward-compatible where possible.

**Revision tracking:** `alembic/versions/` directory with one file per migration.
Revisions are identified by 8-character hashes (e.g., `001_initial_schema.py`).

---

## Migration Workflow

### 1. Creating a Migration

**When to create:**
- Add/remove/rename a table
- Add/remove/rename a column
- Add/remove an index
- Change column type or constraint
- Add RLS policy
- Add/remove a enum value

**Never modify existing migration files** — they're immutable once deployed.

```bash
cd backend
alembic revision --autogenerate -m "Description of change"
```

This creates a new file in `alembic/versions/` like:
```
002_add_stripe_columns.py
```

**Manual edits:** Review the autogenerated file carefully. Sometimes you need to:
- Add `server_default` for non-null columns on existing tables
- Split complex migrations into multiple steps
- Add data migrations (populate new columns)

---

### 2. Review & Commit

**Code review checklist:**
- [ ] Migration is reversible (`downgrade()` implemented)
- [ ] Adding columns: `nullable=True` or `server_default` provided
- [ ] Dropping columns: data will be lost — acceptable?
- [ ] Renaming columns: use `rename_column()` (not drop + add)
- [ ] Indexes: `CONCURRENTLY` for production (PostgreSQL) if table large
- [ ] Long-running migrations: consider `batch_operation()` for large tables

---

### 3. Testing Locally

```bash
# 1. Reset to base (WARNING: destroys data)
alembic downgrade base

# 2. Apply all migrations
alembic upgrade head

# 3. Verify schema
alembic current
alembic history

# 4. Test downgrade
alembic downgrade -1  # step back one
alembic upgrade head  # re-apply
```

For integration tests, migrations run automatically in `pytest` fixtures.

---

### 4. CI/CD Pipeline

On every PR merge to `main`:

1. **Migration lint:** `alembic check --verbose`
2. **SQL generation:** Verify `alembic revision --autogenerate` produces no changes (schema in sync)
3. **Dry-run on test DB:** Apply to fresh Postgres, run full test suite
4. **Production deployment:** Manual approval required for migration steps

## Migration Types

### Schema Migration (DDL)

```python
def upgrade():
    op.add_column('users', sa.Column('phone', sa.String(), nullable=True))
    op.create_index('idx_users_phone', 'users', ['phone'])
```

### Data Migration (DML)

```python
def upgrade():
    # Backfill existing data
    op.execute("""
        UPDATE users 
        SET phone = 'unknown' 
        WHERE phone IS NULL
    """)
```

**Tip:** Separate schema from data for easier rollback.

### Composite Migration

```python
def upgrade():
    # Step 1: Add nullable column
    op.add_column('users', sa.Column('new_field', sa.String(), nullable=True))
    
    # Step 2: Backfill in batches (avoid long locks)
    conn = op.get_bind()
    conn.execute("UPDATE users SET new_field = 'default' WHERE new_field IS NULL")
    
    # Step 3: Set NOT NULL constraint
    op.alter_column('users', 'new_field', nullable=False)
```

## Zero-Downtime Deployments

### Strategy: Expand-Contract (Two-Phase)

**Phase 1 — Expand (Add nullable column):**
```python
# Migration 003_add_email_verified.py
def upgrade():
    op.add_column('users', sa.Column('email_verified', sa.Boolean(), nullable=True))
```
- Application code handles both `NULL` (old) and `False` (new)
- Deploy app with backward-compatible code

**Phase 2 — Contract (Make NOT NULL, backfill):**
```python
# Migration 004_make_email_verified_not_null.py
def upgrade():
    # 1. Backfill
    op.execute("UPDATE users SET email_verified = false WHERE email_verified IS NULL")
    # 2. Set NOT NULL
    op.alter_column('users', 'email_verified', nullable=False)
```
- Deploy app with new logic that assumes column always present

**Phase 3 — Add default (optional)**
```python
# Migration 005_set_email_verified_default.py
def upgrade():
    op.alter_column('users', 'email_verified', server_default='false')
```

**Rollback safety:** At any point, old code (pre-expand) can run against new DB if column is nullable.

## Rollback Procedures

### Automatic Downgrade

Every migration must implement `downgrade()` to reverse changes:

```python
def downgrade():
    op.drop_column('users', 'email_verified')
    op.drop_index('idx_users_email_verified')
```

**Test it:**
```bash
alembic downgrade -1  # Revert last migration
alembic upgrade head  # Re-apply
```

If `downgrade()` fails, manual intervention required.

---

### Emergency Rollback (Production)

**Scenario:** Migration broke something, need immediate revert.

**Option A: Alembic downgrade (fast)**
```bash
kubectl exec -it <backend-pod> -- alembic downgrade 001
```
**Only if:** Downgrade is known-safe and fast (< 30s).

**Option B: Hotfix migration (safer)**
Create new migration that reverses problematic changes:

```python
# 004_fix_broken_column.py
def upgrade():
    # Remove broken column added in 003
    op.drop_column('users', 'broken_column')
    # Re-add correctly
    op.add_column('users', sa.Column('fixed_column', sa.String(255)))
```

Deploy immediately. No need to touch live DB with `downgrade`.

## Long-Running Migrations

### Problem: `ALTER TABLE` locks table (blocking writes)

**Big tables** (> 1M rows): Use `CONCURRENTLY` for index creation:

```python
def upgrade():
    # For PostgreSQL only
    op.create_index('idx_users_email', 'users', ['email'], postgresql_concurrently=True)
```

But `CONCURRENTLY` cannot run inside transaction. Alembic workaround:

```python
def upgrade():
    # Run outside transaction
    bind = op.get_bind()
    bind.execute("CREATE INDEX CONCURRENTLY idx_users_email ON users (email)")
```

**Caution:** If migration fails mid-way, index may be left in `IN PROGRESS` state. Must clean manually.

---

### Batch Updates

Update rows in batches to avoid long transactions:

```python
def upgrade():
    conn = op.get_bind()
    batch_size = 10000
    offset = 0
    
    while True:
        result = conn.execute("""
            UPDATE users 
            SET status = 'active'
            WHERE status IS NULL
            LIMIT %s OFFSET %s
        """, (batch_size, offset))
        
        if result.rowcount == 0:
            break
        offset += batch_size
```

## Migration Best Practices

1. **Always test on staging with production-like data volume**
2. **Add `WHERE` clause to avoid full table scans on large tables**
3. **Use `CONCURRENTLY` for indexes on live tables**
4. **Never drop a column in the same migration you add a replacement** — keep both for at least 1 release cycle
5. **Never change column type in-place** — add new column, copy data, switch over, drop old
6. **Document non-obvious changes in comments above migration**
7. **Set `timeout` on long-running migrations:** `op.execute("SET statement_timeout = 300000")`

## Rollback Playbook

### Step 1: Assess Impact

```bash
# See current revision
alembic current

# See full history
alembic history --verbose

# What will downgrade do?
alembic downgrade --sql 001  # prints SQL only
```

### Step 2: Maintenance Mode (if needed)

If migration might cause prolonged downtime:

```bash
# Put LB in maintenance mode
kubectl patch svc backend -p '{"spec":{"externalTrafficPolicy":"Local"}}'

# Or set readiness probe to fail
kubectl get pod -l app=ai-f-backend
kubectl rollout pause deployment/backend
```

### Step 3: Execute Rollback

**Safe rollback (downgrade 1-2 steps):**
```bash
kubectl exec -it <backend-pod> -- alembic downgrade -1
```

**Full rollback to known-good:**
```bash
kubectl exec -it <backend-pod> -- alembic downgrade 001
```

⚠️ **Warning:** Some downgrades are **destructive** (drop table, delete data). Review `downgrade()` before running.

### Step 4: Verify & Deploy Fix

```bash
# Check DB schema
alembic current
alembic history

# Run smoke tests
pytest tests/smoke/test_basic_api.py

# Deploy hotfix migration if needed (see "Hotfix migration" above)
```

## Common Issues & Solutions

| Issue | Cause | Fix |
|-------|--------|-----|
| `alembic revision` produces empty diff | Models already match DB | OK — no changes needed |
| `duplicate column` error | Migration already applied on dev DB | Reset with `alembic downgrade base` then re-upgrade |
| `cannot drop column` | Column has data and RESTRICT default | Add `DROP ... CASCADE` or backup data first |
| Migration takes hours | No index on WHERE clause column | Add index first in separate migration |
| `Deadlock detected` | Concurrent DDL | Use `CONCURRENTLY`, serialize migrations |
| `downgrade()` fails | Missing reverse data restoration | Manual SQL patch or hotfix migration |

## Production Checklist

Before applying migration to production:

- [ ] Tested on staging with production-scale data (10M+ vectors)
- [ ] `downgrade()` tested and works
- [ ] Estimated runtime < 2 min for normal load (long migrations scheduled in maintenance window)
- [ ] Run during approved maintenance window (Sundays 02:00-04:00 UTC)
- [ ] DB snapshot taken (RDS automated backup is sufficient)
- [ ] Monitoring alerts: DB CPU, connection count, query latency
- [ ] Runbook documented (this document)

---

## Monitoring During Migration

Watch these metrics:

- **PostgreSQL:** `pg_stat_activity` (long-running queries)
- **CPU:** RDS CPU utilization
- **Connections:** Active connections to DB
- **Errors:** `alembic` pod logs for errors

If migration exceeds timeout, **DO NOT CANCEL** — let it finish or implement manual rollback after assessing partial state.

## Migrations Without Downtime

**Golden rule:** Every schema change must be **forward & backward compatible** for at least 1 release cycle.

Compatible changes:
- ✅ Add nullable column
- ✅ Add column with default (but backfill required)
- ✅ Add table
- ✅ Add index `CONCURRENTLY`
- ✅ Rename column using `rename_column()` (PostgreSQL keeps old column until next release)

Unsafe (requires downtime):
- ❌ Drop column (data loss)
- ❌ Change column type (incompatible)
- ❌ Add NOT NULL constraint without default on non-empty table

**Release pattern for NOT NULL:**
```
v1.0:   Column nullable, code handles NULL
v1.1:  Backfill all NULLs → known value
v1.2:  Set NOT NULL constraint (safe now)
```
